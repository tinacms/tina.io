## What's new

Across May and June the team merged 70+ PRs into TinaCMS and shipped a wave of updates to TinaCloud. Here are the highlights:

### ⭐️ TinaCMS in the Astro news

TinaCMS got a shout-out in [Astro's "What's new in May" post](https://astro.build/blog/whats-new-may-2026). We here have loved trying out Astro in our own projects and have been impressed with what their team has put out!

### 🧩 All-in on Astro

**This is a big change for us.**

TinaCMS visual editing no longer needs React.

A \~2 KB vanilla-JS bridge can now power live editing, and it now works on statically-built pages too, with Astro 6 supported and the starter rebuilt React-free.

We've done this as Astro was winning in terms of user interest – we've been collecting some stats on this. After this update Astro jumped up to **6× the starter clone rate of Next.js** for our CMS starter projects!

We've kept polishing the Astro dev experience too — new pages now show up as you write content, without having to restart the dev server.

### 🚀 Tina V4 announced

We officially announced [Tina V4](/blog/tinacms-v4-is-coming). The seven-epic foundation work kicked off in Sprint 109, and auth is heading to WorkOS AuthKit.

And it's not just an announcement — the groundwork is already landing. We've scaffolded the new core package and shipped the first of the V4 field plugins (String, Boolean and Number), along with early docs for the new plugin system.

### 🩺 Upgrade-path tooling

Two changes to smooth the road to V4: the new `tinacms doctor` command flags any outdated Tina package against the latest on npm, and legacy `.tina/` folders now surface a deprecation warning. We're hoping this makes the maintenance story of using TinaCMS more compelling.

### 🔒 Security hardening

We ran a focused security pass over the codebase. The dev server now rejects cross-origin requests that try to change state, message origins and rich-text URLs are validated and sanitised, and every GitHub Actions workflow runs with least-privilege permissions. We also locked down our npm publish pipeline and added dependency review, with a cooldown before new package versions are pulled in.

None of this is something you have to think about — it just makes Tina safer by default.

### 🪶 A lighter admin bundle

We trimmed weight that editors never needed to download. Swapping `moment` for `date-fns` and clearing out some legacy code shaved around 40 KB off the shipped admin bundle, so the editing UI loads a little faster.

### ✨ Editing experience

A few quality-of-life upgrades for editors: the editorial workflow now has a save-options dropdown for choosing between draft and ready-for-review, and admin editor pages have a back-to-collection breadcrumb so it's quicker to move around. `create-tina-app` also shows live install activity as it scaffolds, so you're not left watching a silent spinner.

### 🧪 More tests

We're committed to writing more frontend tests, to improve reliability and limit any risk of regressions.

### 📦 Downloads

Over **200,000** package downloads in the month of May.

### ☁️ TinaCloud Improvements

The cloud team worked on reliability, and the numbers reflect that:

* **Content API 500s cut from \~13K to \~2K**, and **Identity API 500s down 87%**.
* Search rebuilds on large repos no longer fail from GitHub rate limits.
* Rate limiting now stops runaway traffic from taking the API down.

We also shipped **Separate Content Repos**: you can now bind two GitHub repos to a single TinaCloud project — one for your app code, one for your editorial content — all under one dashboard. More in [Separate Content Repos are here for TinaCloud](/blog/tinacloud-separate-content-repos).
