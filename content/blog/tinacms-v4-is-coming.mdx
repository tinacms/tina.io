Today we're sharing some big news: work is about to begin on **TinaCMS v4**.

As TinaCMS has grown over the last few years, we've spotted areas where the foundations could work harder for the people building on top of them. v4 is our chance to slow down, reshape those foundations, and make TinaCMS far easier to extend, customise, and grow with.

This post is a first look at what's coming.

## A New Package, A Fresh Start

TinaCMS v4 will ship as a brand new package: `@tinacms/tinacms`.

We're taking the same approach Angular took when it moved from AngularJS to Angular 2+ — shipping the new version under a different package name so that v3 can keep doing its job while v4 is built on modernised foundations. Existing projects can continue to rely on the current `tinacms` package without any disruption, and adopting v4 will be an explicit, intentional step you take when you're ready.

## A More Extensible Architecture

The biggest shift in v4 is that almost every part of the CMS - fields, sidebars, screens, media providers, search providers becomes a customisable building block you can plug in, swap out, or extend.

What this means for you:

* **Make TinaCMS truly your own.** Build your own field types, sidebar tools, search backends, and media managers as native modules of the system, without having to fork the project.
* **Pick what you need.** Integrations like TinaCloud, Cloudinary, S3, and Algolia move into their own optional building blocks. You pull in only the ones your project actually uses.
* **Smoother upgrades.** Cleaner boundaries between parts of TinaCMS mean updating one area is far less likely to cause unexpected ripples elsewhere.

Every building block will ship with documentation designed to be useful to both developers and AI coding assistants, making it dramatically faster to learn, customise, and integrate TinaCMS into modern AI-assisted workflows.

## Editorial Workflow, Levelled Up

Editorial Workflow has been one of TinaCMS's most-loved features for a while now, a way for content teams to make changes through branches and pull requests without leaving the editor.

In v4, it stops feeling like a feature added on the side and becomes a deeply integrated part of the editing experience. Expect:

* Smoother branch creation and switching
* Pull request titles, descriptions, and status surfaced directly inside the editor
* A workflow that feels natural even if you've never touched Git before

If you already love Editorial Workflow, you're going to love it even more.

## A Rebuilt Forms System

Forms are the surface you spend the most time with, so we're giving them the attention they deserve.

* **A modern forms engine.** We're moving from Final Form to React Hook Form - a faster, actively maintained library with a much larger ecosystem.
* **Reliable dirty/pristine tracking.** No more guessing whether your changes have been saved.
* **Unsaved-change warnings.** Navigate away with confidence - the editor will let you know if there's work you haven't saved.
* **Drafts and stronger validation.** Save work in progress, and catch issues at the field level, including custom rules and validation that spans multiple fields.
* **Flat mode for nested content.** Edit deeply nested components on a single page instead of drilling through layers of navigation.
* **More control points for developers.** New hook points around save and edit give your code precise control over what happens at each stage of the editing pipeline.

## A Refreshed Editor UI

The TinaCMS interface is getting a major UX overhaul - a restructured layout, redesigned navigation, and a cleaner, more consistent look that keeps the TinaCMS branding you know.

You can expect:

* A reorganised admin panel with clearer navigation between collections, media, and editor views
* A redesigned media manager with improved folder navigation
* A more polished field editing experience, including new panel and window modes for working with complex content
* Better array and object editing, with flat layouts, copy and delete buttons, and breadcrumbs to keep you oriented in nested data

## Fields You Can Trust

Every field type - text, numbers, dates, images, references, rich text, and the rest is getting the same care and attention in v4.

What that means in practice:

* **More consistent behaviour.** No more surprises when one field type acts differently from another.
* **Easier validation and customisation.** Add your own rules without fighting the system.
* **Better support for custom fields.** Teams that build their own field types will find them much simpler to create, share, and maintain.
* **Familiar to use, stronger underneath.** The fields you use every day will feel like the ones you know, but built on a much more solid base.

## A Painless Migration Path

We know that any major version bump raises the same question: *"What does this mean for my project?"*

Our goal is straightforward. **For the vast majority of projects, upgrading to v4 should be as simple as installing the new package and running a codemod**.

* A purpose-built codemod will handle the schema migration automatically.
* Migration documentation will cover every breaking change, what the codemod takes care of, and what (if anything) needs manual attention.
* Projects with custom fields or heavily customised components will get clear before/after examples to follow.
* Troubleshooting guides will walk through common edge cases.

If your project uses a fairly standard TinaCMS setup, the upgrade should take minutes, not days.

## v3 Is Going Nowhere

To be clear: **TinaCMS v3 is not being deprecated overnight**. It will continue to be supported while v4 is built and stabilised. You can stay on v3 as long as you need to, and migrate when the timing is right for your project.
