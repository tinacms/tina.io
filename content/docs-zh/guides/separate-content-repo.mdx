## 介绍

Tina支持从一个独立的Git仓库获取内容。通过这种方法，你可以继续在你的“网站仓库”中定义Tina配置，但文档将会保存在其他地方。

### 为什么？

你可能出于多种原因想要这样做，例如：

* 将你的网站代码与内容文件解耦，这样你的提交历史就不会混合内容更新和代码更新
* 避免必须给予TinaCloud对你网站代码的写入权限（只需给予对内容仓库的访问权限）

  </>}

### 要求

* 你必须在本地机器上同时检出“内容仓库”和“网站仓库”。
* 你必须在Tina配置中提供内容仓库的位置（下面会详细介绍）。
* 在进行更改时，你需要确保已将*两个*仓库推送到GitHub，且“内容仓库”的更改优先
* 在进行更改时，你需要在网站仓库和内容仓库之间同步`tina/`中的生成文件（如`tina-lock.json`、`__generated__`文件等）

## 创建网站仓库

我们已经部署了一个基本的起始“网站仓库”，你可以克隆它以开始。

```sh
git clone git@github.com:tinacms/separate-website-repo.git
```

“网站仓库”包含NextJS站点和`tina`配置。

## 创建内容仓库

要设置“内容仓库”，我们将从一个简单的`.mdx`文件开始：

```sh
mkdir -p ../demo-content-repo/content/pages && touch ../demo-content-repo/content/pages/home.mdx && echo "Hello" >> ../demo-content-repo/content/pages/home.mdx
```

此命令创建了一个文件夹（`demo-content-repo`）并在`content/pages`目录中添加了一个MDX文件，我们将在Tina配置中使用该目录定义`page`集合。

## 本地开发

在“网站仓库”中，安装项目的依赖项

```sh
yarn install
```

在本地运行项目：

```sh
yarn dev
```

## 在编辑模式下访问页面

打开`http://localhost:3000/admin/index.html`。

从这里你可以在`tina/config.js`中为你的内容模型添加更多字段。[访问文档](https://tina.io/docs/schema/)以了解更多关于内容建模的信息。

## 部署你的内容仓库

接下来，我们将从“内容仓库”创建一个TinaCloud项目

1. 在你的内容仓库中初始化Git并推送到GitHub。
2. 从那里[连接到TinaCloud](https://tina.io/docs/tina-cloud/dashboard/projects/)
3. 完成后，为`main`分支[创建一个令牌](https://tina.io/docs/tina-cloud/dashboard/projects/#api-tokens)，或使用`*`允许所有分支。

切换回你的“网站仓库”并设置一个`.env`文件以在连接到TinaCloud时使用：

```
cp .env.example .env
```

当你运行`tinacms build`时，它将使用这些凭据连接到TinaCloud而不是本地服务器：

```sh
yarn build
```

## 管理跨仓库的架构更改

当你修改Tina架构（tina/config.(ts/js)）

1. 运行tinacms build以重新生成文件 - Tina将重新生成文件，如你的tina-lock.json
2. 提交并推送这些更改到你的网站仓库和内容仓库，以确保跨构建的兼容性

## 路径配置注意事项

在你的tina/config.(ts/js)中定义集合路径时，避免设置path: ''。因为Tina期望一个相对于文件夹的路径。使用''可能会导致在创建新文档时出现错误，因为Tina在保存时会添加一个/，导致无效的GitHub路径。

## 静态网站生成器兼容性

如果你使用的是Next.js以外的静态网站生成器（SSG），如Astro。确保你的内容管道忽略内容仓库中的tina/目录。Tina的配置文件不是内容文件，这可能会导致解析错误。

```javascript
export default defineCollection) {
  schema: mySchema,
  exclude: ['tina/**/*'],
};
```
